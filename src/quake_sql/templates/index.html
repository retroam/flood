Overview

Quake SQL turns plain-English questions into live ClickHouse queries against the USGS earthquake feed. The app now has two runtime loops: a query path that generates and executes SQL for a user question, and an eval path that runs the same benchmark cases twice for the currently configured model, once with CFG and once without it. That makes the eval tab a direct ablation of the grammar constraint instead of a static multi-model report.

Why Grammar Constraints?

flowchart TB
    subgraph Traditional["Traditional Text-to-SQL"]
        P1["Prompt\nEngineering"] --> O1["Free-form\nLLM Output"]
        O1 --> P2["Post-hoc\nParsing"]
        P2 -- "Malformed SQL" --> ERR["Runtime\nError"]
        P2 -- "Valid syntax" --> OK1["Execute"]
    end

    subgraph Constrained["Grammar-Constrained (This App)"]
        P3["Prompt +\nLark Grammar"] --> O2["Constrained\nDecoding"]
        O2 --> V2["Semantic\nValidation"]
        V2 -- "Wrong columns" --> O2
        V2 -- "Valid" --> OK2["Execute"]
    end

    style ERR fill:#fee2e2,stroke:#ef4444,color:#1d1c1d
    style OK1 fill:#d4edda,stroke:#2eb67d,color:#1d1c1d
    style OK2 fill:#d4edda,stroke:#2eb67d,color:#1d1c1d
    style O1 fill:#fff3cd,stroke:#f59e0b,color:#1d1c1d
    style O2 fill:#d4edda,stroke:#2eb67d,color:#1d1c1d
  • Syntactically valid by construction -- the model cannot produce tokens that violate the grammar.
  • Injection-proof -- prompt manipulation cannot escape the grammar's structural constraints.
  • Validator focuses on semantics -- only checks column names and safety invariants, not syntax.
  • A/B evals stay honest -- the benchmark now measures the effect of CFG directly by running the same cases with and without the grammar constraint.
